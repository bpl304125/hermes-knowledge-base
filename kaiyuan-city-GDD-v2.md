## 第一章 项目概述

### 1.1 项目简介与目标
公元2045年，"记忆合金"技术可复现古代建筑。你作为时空规划师，从村落起步，解锁唐宋元明清各朝建筑，建设跨越千年的盛世之城。

- **平台**: Steam (PC) + App Store/Google Play + 抖音小游戏

### 1.3 目标用户与市场分析
- **核心用户**: 18-35岁，热爱模拟经营/策略，对中国文化有兴趣
- **竞品**: 《城市天际线》《江南百景图》《模拟城市》

## 第二章 游戏设计核心理念

### 2.1 世界观与背景设定

**故事梗概**: 2045年，记忆合金技术问世。玩家作为"时空规划师"，从一块空地开始，通过合金"重现"各朝代经典建筑。从汉代市坊到唐代大明宫，从宋代虹桥到清代四合院——一座跨越千年的城市在你手中诞生。

**主要角色**:
- 🏛️ **玩家(时空规划师)**: 拥有记忆合金技术
- 👴 **老陈**: 村民代表，传统建造顾问
- 🤖 **AI小元**: 城市管理AI，数据分析与优化建议
- 🏯 **历代名匠**: 解锁各朝代后出现的NPC（宇文恺、李诫、样式雷）

**叙事架构**: 沙盒为主 + 里程碑事件触发剧情

### 2.2 核心玩法设计

**核心循环**: 获取资源 → 建造建筑 → 吸引人口 → 增加税收 → 解锁新内容 → 扩大城市

**规则机制**:
- 7维资源: 💰金币 👥人口 ⚡电力 💧水源 😊满意度 🚢贸易 🏛️文化
- 建筑分类建造，工业远离住宅；国风建筑需人口门槛
- 满意度<0 → 人口流失 → 经济崩溃

**胜负条件**: 沙盒无强制胜负。终极成就：「盛世之城」(人口10万+地标全解锁)。

### 2.3 互动机制与玩家旅程

**关卡结构**: 3个教学关(道路→住宅→水电) → 自由模式 → 挑战模式

**特色机制**:
- ☄️ 灾难系统: 陨石/洪水/火灾/地震，主动触发改建
- 🏯 时代演进: 人口达阈值解锁下个朝代建筑群
- 🚢 全球贸易: 港口→海外航线→稀有资源+地标碎片
- 📱 好友拜访: 参观好友城市获取灵感奖励

## 第三章 开发流程规划

游戏开发按照行业通行做法分为五个核心阶段，各阶段目标明确、环环相扣。

### 3.1 立项阶段 (Week 1-2)

**3.1.1 头脑风暴会议**
- 全体成员提出创意方向: 国风建造 / 科幻策略 / 休闲模拟
- 集体讨论后投票确定「国风城市模拟经营」
- 产出: 会议纪要 + 创意投票结果

**3.1.2 策划草案(意向书)**
- 说明游戏基本构想: 3D城市建造 + 国风建筑 + 时代演进
- 核心玩法概述: 资源管理→建筑→人口→税收循环
- 产出: 一页纸意向书

**3.1.3 四维可行性评估**

| 维度 | 评估 | 结论 |
|------|------|------|
| 市场 | 模拟经营品类需求稳定，国风差异化明显 | ✅ 可行 |
| 核心玩法 | 城市建造+资源管理经市场验证 | ✅ 可行 |
| 内容 | 6个月可完成核心内容 | ✅ 可行 |
| 变现 | Steam买断+移动端IAA，双线收入 | ✅ 可行 |

### 3.2 预制作阶段 (Week 3-6)

**3.2.1 技术框架搭建**
- 评估 Three.js vs Godot vs Unity
- 选型决策矩阵: Godot 4 (开源/轻量/2D3D混合)
- 产出: 技术选型文档

**3.2.2 可玩性原型 (Prototype)**
- ✅ 已完成: Web版可玩原型
- 包含: 8种建筑 + 7维资源 + 视角控制 + 灾难系统
- 反馈: 核心循环成立，需优化经济数值

**3.2.3 早期市场调研**
- 同类产品分析: 《城市天际线》Steam评价92%好评 / 《江南百景图》移动端月流水千万级
- 产出: 《市场调研报告》包含竞品数据、用户画像、定价策略

**3.2.4 核心要素确定**
- 核心玩法: 建造→经营→扩张循环 ✅
- 世界观: 2045记忆合金+国风建筑 ✅
- 商业化: PC买断制(¥36) + 移动端免费+IAA + 抖音小游戏广告

### 3.3 正式制作阶段 (Month 2-5)

**按前/中/后期规划，Sprint双周迭代交付**:

```
前期 (Month 2):   核心建造系统 + 基础经济 + 5种建筑
中期 (Month 3-4): 全部建筑 + 资源系统 + 国风建筑群 + 时代演进
后期 (Month 5):   灾难系统 + 贸易 + 好友 + UI打磨 + 音效
```

**Sprint节奏 (2周)**:
```
Sprint Planning (周一) → Daily Standup (15min) → Sprint Review (周五) → Retro
每个Sprint产出: 可运行Build + 变更日志
```

**里程碑版本**:
| 里程碑 | 时间 | 交付物 | 验收标准 |
|--------|------|--------|---------|
| M1 Prototype | Week 6 | 可玩原型 | 30分钟不崩溃 |
| M2 Core | Week 10 | 完整建造+经济 | 100人口稳定运行 |
| M3 Heritage | Week 14 | 国风建筑+时代 | 解锁3朝代 |
| M4 Systems | Week 18 | 灾难+贸易闭环 | 所有系统功能完整 |
| M5 Alpha | Week 22 | Alpha测试版 | 50人内部测试 |
| M6 Release | Week 26 | 发布版本 | 零P0 Bug |

### 3.4 测试阶段 (Month 5-6)

**3.4.1 Alpha测试 (内部)**
- 范围: 开发团队 + 10名内部测试者
- 重点: 功能完整性验证
- 指标: 完成率>80%，崩溃率<1次/小时
- 工具: 手动测试 + 自动化冒烟测试

**3.4.2 Beta测试 (外部)**
- 范围: 50名外部玩家
- 重点: 留存数据 + 基础通关率
- 核心指标:
  - D1留存 > 50% (次日再玩)
  - D7留存 > 25%
  - 教程完成率 > 90%
  - 首局时长 > 10分钟
- 反馈渠道: Discord + 游戏内问卷

**3.4.3 质量评审**
- 功能测试: 全功能用例覆盖
- 性能测试: Tracy/RenderDoc 帧分析，目标30fps@移动端
- 兼容性测试: 5+种Android设备 + 3种iOS设备
- CE测试(用户验收): 最后4周持续收集反馈

### 3.5 发行阶段 (Month 6+)

**3.5.1 市场推广计划**
```
发布前90天: Steam商店页上线 → 愿望单开放
发布前60天: 首支预告片 → Discord社区建立
发布前30天: Steam Next Fest (带Demo) → 媒体Key分发
发布前14天: YouTuber/B站UP主合作 → 社交预热
发布前7天:  最终预告 → 倒计时活动
发布日:      首发9折 → 72h紧急热修待命 → 玩家反馈收集
```

**3.5.2 抖音运营专项**
- 短视频策略: 录屏→精彩时刻剪辑→15-30秒发布
- 内容方向: 城市建造前后对比 / 灾难事件 / 国风建筑展示
- 频率: 日更1-2条

**3.5.3 上线后运营**
- 第一周: 每日Hotfix
- 第一月: 周更新(修Bug+小优化)
- 长期: 双周迭代 + 季度大更新(新朝代/新建筑/新活动)
- 反馈闭环: 玩家反馈→优先级排序→Sprint规划→交付

## 第四章 团队分工与协作

### 8.2 目标达成对比

|---------|---------|------|

| 团队组建 | ⏳ 单人开发中 | 待解决 |

### 8.3 经验教训

**可复用的做法**:
1. 先Web原型验证玩法 → 再选型正式引擎，避免了过早技术绑定
2. GDD与代码同步更新，文档不滞后
3. 持续学习→知识库沉淀→即时应用

**待改进问题**:
1. 经济数值需要更多Excel模拟，避免"边写代码边调数值"
2. 性能测试应在原型阶段就开始，而非等到正式制作
3. 用户测试反馈收集不够系统化

### 8.4 心得反思

**技术方面**: Web原型验证了核心循环可行，但移动端性能是最大挑战。Godot 4的2D/3D混合能力适合本项目。

**团队协作**: 单人开发效率有限，M2后需至少1名程序+1名美术。

**项目管理**: 敏捷Scrum适合游戏开发，但要严格控制Sprint范围，2周迭代节奏合理。
