# 游戏网络与音频系统深度研究

---

## 一、多人游戏网络架构

### 1.1 客户端预测 (Client-Side Prediction)

**问题**: 按W键→发送到服务器→服务器验证→返回位置→移动角色，200ms RTT导致操作延迟不可接受。

**解决方案**: 客户端立即执行预测
```
客户端输入 → 立即移动(预测) → 同时发送到服务器
服务器: 接收输入 → 权威模拟 → 广播正确位置
客户端: 收到服务器状态 → 比较差异 → 和解(reconciliation)
```

### 1.2 服务器和解 (Server Reconciliation)

```
客户端保存历史输入序列 [seq=100,seq=101,seq=102...]
收到服务器状态(附带lastProcessedSeq=100):
├── 如果预测位置 ≈ 服务器位置: 保持(预测正确)
├── 如果差异大: 回滚到seq=100的状态 → 重新应用seq=101,102输入
└── 平滑插值到修正位置(避免跳变)
```

### 1.3 实体插值 (Entity Interpolation)

**问题**: 服务器以20Hz发送其他玩家快照，需平滑显示。

```
插值延迟 = 2 × 服务器Tick间隔(100ms)
当前渲染状态 = lerp(snapshot[N-2], snapshot[N-1], fraction)
```

### 1.4 延迟补偿 (Lag Compensation)

**问题**: 高延迟玩家射击屏幕上的敌人，但服务器上敌人已移动。

```
服务器端回溯:
1. 接收射击事件(userCmdTime = T0)
2. 回滚所有玩家状态到T0时刻
3. 在T0状态执行射线检测
4. 恢复当前状态
5. 判定命中 → 伤害结算
```

### 1.5 回滚网络 (Rollback Netcode)

**格斗游戏 GGPO 核心思想**:
```
固定帧步长 + 推测执行 + 回滚修正

帧N-1: 双方等待输入
帧N:   没有对方输入 → 推测(按上次输入) → 继续执行
帧N+1: 收到对方输入 → 回滚到帧N → 重新模拟 → 显示正确状态

最高可容忍~200ms延迟的流畅体验
```

### 1.6 MMO 兴趣管理 (Area of Interest)

```
空间分区方案:
├── 静态网格: 世界划分为固定格子，只同步相邻格子
├── 动态AOI: 以玩家为中心半径R的圆形区域
├── 四叉树/八叉树: 层级空间管理
└── 可见性: 视野锥+距离双重过滤

优化: 
- 远距离: 低频更新(1Hz) + 低精度
- 中距离: 中频(5Hz)
- 近距离: 高频(20Hz) + 完整状态
```

### 1.7 网络优化技术

| 技术 | 原理 | 节省 |
|------|------|------|
| **快照压缩** | 整数化+delta编码+字典+LZ4 | 80-90% |
| **增量更新** | 只发送改变的状态字段 | 50-70% |
| **兴趣管理** | 不发送看不见的实体 | 与可见实体数成正比 |
| **可靠vs非可靠** | 关键状态TCP/位置UDP | 按需 |
| **对象池化** | 避免网络对象频繁创建/销毁 | 内存稳定 |

### 1.8 开源网络库

| 库 | 特点 | 适合 |
|------|------|------|
| **ENet** | C、轻量、可靠UDP | 独立/小型多人 |
| **SteamNetworkingSockets** | Valve开源、P2P+中继 | Steam游戏 |
| **GameNetworkingSockets** | 支持ICE/STUN打洞 | P2P游戏 |
| **Netcode.IO** | Unity专注 | Unity项目 |
| **Mirror** | Unity高级网络框架 | Unity多人 |

## 二、游戏音频系统

### 2.1 音频中间件对比

| | FMOD | Wwise | 引擎内置 |
|------|------|-------|---------|
| 授权 | 预算<$200K免费 | 预算<$200K免费 | 随引擎 |
| 工具 | FMOD Studio | Wwise Authoring | 有限 |
| 动态音乐 | ✅ 强大 | ✅ 强大 | 基础 |
| 空间音频 | ✅ | ✅ | 基础 |
| 平台 | 全平台 | 全平台 | 随引擎 |
| 学习曲线 | 中 | 高 | 低 |

### 2.2 空间音频 (Spatial Audio)

```
3D音频定位线索:
├── ITD (双耳时间差): 声音到达左右耳时间差
├── ILD (双耳强度差): 头部遮挡产生音量差
├── HRTF (头相关传递函数): 模拟耳廓/头部/躯干对声波的滤波
└── Distance Attenuation: 距离衰减 + 空气吸收

Ambisonics (全景声):
- 一阶(4通道): WXYZ基本方向
- 三阶(16通道): 精细空间定位
- 旋转: 简单的矩阵变换，无需重计算
```

### 2.3 音频遮挡与衍射

```
发射体 → [直射路径] → 接收者
发射体 → [绕射路径] → 接收者(绕过障碍)
发射体 → [透射路径] → 接收者(穿过墙壁)

实时计算:
射线检测 → 无遮挡: 直射音量100%
射线检测 → 有遮挡: 
  ├── 衍射路径查找(最短绕行路径)
  ├── 透射衰减(根据墙壁材质: 木板-6dB, 砖墙-20dB, 混凝土-40dB)
  └── 低通滤波(障碍越大，高频衰减越多)
```

### 2.4 动态音乐系统

**水平重混 (Horizontal Re-sequencing)**:
```
音乐片段: Intro → A段 → B段 → A段 → C段 → Outro
根据游戏状态动态排序: 
  低紧张度 → A段(安静)
  中紧张度 → B段(节奏加快)
  高紧张度 → C段(高潮)
```

**垂直分层 (Vertical Layering)**:
```
层1: 基础打击乐   (始终播放)
层2: 贝斯线       (紧张度>20%加入)
层3: 和弦         (紧张度>40%加入)
层4: 旋律         (紧张度>60%加入)
层5: 高潮弦乐     (BOSS战100%)
混合: 交叉淡入淡出各层音量
```

**过渡设计**:
- 瞬切(Stinger): 短音效桥接两个段落
- 小节对齐: 等当前小节结束再切换
- 节拍对齐: 等当前节拍结束

### 2.5 程序化音频生成

| 技术 | 原理 | 应用 |
|------|------|------|
| **减法合成** | 振荡器→滤波→包络 | 科幻音效、引擎声 |
| **粒子合成** | 微小声音片段(粒子)实时排列 | 环境音/质感 |
| **物理建模** | 模拟弦/管/膜振动方程 | Karplus-Strong弹拨 |
| **波表合成** | 预存波形→动态混合 | 音乐合成 |
| **频谱合成** | FFT分析→频谱操作→逆FFT | 变形音效 |

### 2.6 音效设计分层

```
总混音(Master Bus):
├── 音乐总线(Music Bus)
│   ├── 动态音乐控制器
│   └── 环境音乐
├── 音效总线(SFX Bus)
│   ├── UI音效(点击/确认/提示)
│   ├── 角色音效(脚步/武器/技能)
│   └── 环境音效(风/雨/动物)
├── 语音总线(Voice Bus)
│   ├── 对话
│   └── 旁白
└── 环境总线(Ambience Bus)
    └── 持续背景音

每总线独立:
- 音量/混响/EQ/压缩/侧链闪避(Ducking)
```
