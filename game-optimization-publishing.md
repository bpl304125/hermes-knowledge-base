# 游戏优化、发布与运维

---

## 一、移动端优化 (抖音小游戏核心)

### 1.1 包体优化

```
目标: 首包<4MB，总包<20MB

纹理压缩:
├── Android: ETC2 (GLES 3.0+)
├── iOS: ASTC 4x4/6x6
├── 不要用: PNG/JPG（运行时需解压为RGBA，内存×4倍）
└── 选择: ETC2/ASTC直接上GPU，零解压开销

音频压缩:
├── BGM: Ogg Vorbis q3-q5 (压缩10:1)
├── 短SFX: MP3 96kbps mono
├── 长音频: 流式加载，不驻留内存
└── 格式优先级: mp3 > ogg (兼容性)

字体:
├── 使用系统字体(零包体)
├── 自定义字体: 裁剪到使用字符集(2500→500字符)
└── SDF字体(清晰可缩放，单一纹理)

代码:
├── 引擎裁剪: 去掉不用的模块(3D/物理/UI)
├── 死代码消除: Tree Shaking
└── 压缩: 代码压缩+混淆
```

### 1.2 内存优化

| 策略 | 原理 | 节省 |
|------|------|------|
| **纹理图集** | 合并→减少DrawCall+内存碎片 | 20-40% |
| **MipMap关闭** | 纯2D UI不需要远距离降采样 | 33%纹理内存 |
| **纹理压缩** | ASTC/ETC2替代RGBA | 75%+ |
| **音频流式** | 不预加载长音频 | 5-20MB |
| **引用计数释放** | 切换场景释放旧资源 | 场景级 |
| **对象池** | 复用GameObject | 减少GC |
| **LOD网格** | 远处用低面数 | Mesh内存-60% |

### 1.3 Draw Call优化

```
目标: <100 Draw Call (移动端)
目标: <50 Draw Call (抖音小游戏)

方法:
├── 纹理图集: 同材质→自动批处理→减少DrawCall
├── 静态批处理: 不动的物体标记Static→引擎合并Mesh
├── GPU Instancing: 相同Mesh+Material批量绘制
├── 减少材质种类: 5-10种材质覆盖90%场景
├── 关闭阴影: 移动端阴影DrawCall昂贵
└── UI合批: 同图集UI元素自动合批
```

### 1.4 帧率目标

```
60 FPS: 16.67ms/帧 — 动作/竞技游戏
30 FPS: 33.33ms/帧 — 休闲/三消
自适应: 低电量模式降30fps

每帧预算分配 (16.67ms目标):
├── 物理: 2-3ms
├── AI/逻辑: 2-3ms
├── 渲染: 6-8ms
├── UI: 1-2ms
└── 余量: 2-3ms (预留峰值)
```

## 二、PC/主机优化

### 2.1 渲染优化进阶

```
遮挡剔除:
├── Umbra(PVH): 预计算可见性
├── GPU Occlusion Query: 上一帧结果预测本帧
├── HZB遮挡查询: Hierarchical Z-Buffer加速
└── Portal/Cell: 室内场景分区

光线追踪优化 (RTX):
├── 降噪: NRD实时降噪(OptiX Denoiser)
├── 1/4分辨率: 追踪1/4像素→共享结果
├── 累积: 多帧累加+重用
├── 限制表面: 只对特定材质/距离开RT
└── 混合: 光栅化主画面+RT反射
```

### 2.2 流式加载 (开放世界)

```
分块世界:
世界→16×16 Chunk网格
├── 玩家当前位置周围 3×3 Chunk 高精度加载
├── 5×5 Chunk 中等LOD
├── 更远 Chunk 最低LOD/卸载
└── 异步加载线程: 不阻塞主线程

预加载预测:
- 玩家移动方向预测
- 加载玩家前方Chunk(当前1-2, 前方3-5)
- 后台线程优先级: 前方 > 侧面 > 后方
```

### 2.3 多线程架构

```
主线程: 输入 → 游戏逻辑 → 提交渲染命令
渲染线程: 命令队列 → GPU提交 → 等待完成
工作线程池(4-8线程): 物理/AI/寻路/加载/音频
异步任务: 文件IO/网络/资产解码

Unity DOTS Job System:
[Burst编译的Job]
PhysicsJob (可并行) → AIJob (依赖Physics结果) → RenderJob
```

## 三、发布流程

### 3.1 Steam发布清单

```
发布前90天:
□ 创建Steamworks账号($100)
□ 设置商店页面(预告片+截图+描述)
□ 至少5张高质量截图(1280×720)
□ 至少1个预告片(30-60秒)
□ 愿望单开放

发布前30天:
□ Steam Next Fest报名(带Demo)
□ 构建审核版本
□ 成就系统实现
□ Steam云存档集成

发布前14天:
□ 最终审核提交
□ 媒体Key分发
□ Discord/社交媒体预热

发布日:
□ 设置折扣(首发9折)
□ 监控崩溃报告
□ 社区管理(Steam论坛)
□ 72小时紧急热修准备
```

### 3.2 抖音小游戏发布

```
审核要点:
□ 首包<4MB
□ 无外部链接/二维码(除广告)
□ 无敏感内容
□ 隐私政策+用户协议
□ 激励视频广告位正常
□ 无恶意扣费/诱导分享
□ 性能达标(无频繁ANR/崩溃)
□ 适龄提示(未成年人保护)

发布后运营:
□ 每日数据监控(DAU/留存/广告收入)
□ A/B测试广告位+变现策略
□ 周更新(修复+小内容)
□ 短视频运营: 录屏素材→剪辑→发布
```

### 3.3 CI/CD 流水线

```
Git Push → 
  自动构建(Win/Mac/Linux/Android/iOS) → 
    自动测试(单元/集成/冒烟) → 
      自动打包(Steam/APK/IPA/抖音) → 
        自动上传(分发平台)

工具:
├── GitHub Actions / GitLab CI / Jenkins
├── Unity Build Automation / Unreal Horde
├── Fastlane(iOS/Android自动发布)
└── Docker构建环境统一
```

## 四、线上运维

### 4.1 崩溃监控

```
工具:
├── Sentry/Crashlytics: 实时崩溃报告
├── Unity Cloud Diagnostics
├── 自定义上报: 捕获Critical异常→上传堆栈+玩家日志

崩溃优先级:
P0(紧急): 启动崩溃/支付崩溃/主流程阻塞
P1(高): 特定机型崩溃率>5%
P2(中): 非主流程偶发崩溃
```

### 4.2 热更新

```
方案对比:
├── Lua热更: xlua/tolua，逻辑热更
├── ILRuntime: C#热更，Unity常用
├── HybridCLR: 原生C#热更(推荐)
├── Addressables: Unity资源热更
└── AssetBundle: 传统资源分包

抖音小游戏:
- 代码: 分包加载(≤4MB/包)
- 资源: 远程CDN加载
- 注意: 首场景资源必须在主包
```

### 4.3 运营指标

```
核心指标(北极星):
├── DAU(日活): 每日独立用户
├── 留存: D1(40%+)/D7(20%+)/D30(10%+)
├── LTV(用户终生价值): 总收益/总用户
├── CPI(单用户获取成本): 广告费/新增用户
└── LTV/CPI > 3: 可持续盈利

抖音小游戏指标:
├── 次留(次留率): 次日再玩比例
├── 广告填充率: 广告请求成功率
├── eCPM: 每千次展示收益
├── 人均广告次数: 每日广告观看次数
└── 分享率: 录屏分享比例(裂变核心)
```

### 4.4 反外挂

```
客户端:
├── 代码混淆+加密
├── 反调试检测
├── 内存保护(关键数值加密)
├── 完整性校验(hash验证)

服务器端:
├── 行为分析: 异常移动速度/伤害/获取速度
├── 服务器权威: 所有关键计算在服务器
├── 数据校验: 收益/经验值/物品合理性
└── 举报系统: 玩家举报→人工审核→机器学习自动标记
```
